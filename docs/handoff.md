## 1. 接手时先记住这四件事

1. 公开仓库：[Stephen-creater/upkuajing-tape-masking-film-leads](https://github.com/Stephen-creater/upkuajing-tape-masking-film-leads)
2. 唯一客户总表：[客户表（7.17）.xlsx](../deliverables/客户表（7.17）.xlsx)
3. 当前成果：270 家真实贸易相关公司，全部至少拥有一种状态为 `1` 的有效邮箱或有效电话。
4. 项目当前完成的是“批量邮件之前的客户数据工程”，尚未实现正式群发、退信处理和退订闭环。

对外推荐名称：**海外客户线索获取与批量邮件前置数据工程**。

一句话介绍：

> 这个项目通过跨境魔方 OpenAPI 从海关贸易记录中寻找真实采购过目标产品的公司，经过严格产品匹配、公司去重、联系方式获取和有效性验证，最终形成一份一行一公司的 Excel 客户主表，为后续合规的批量邮件触达提供可靠数据。

## 2. 项目解决什么问题

普通搜索引擎或名单采购通常只能回答“网上有没有这家公司”，无法同时回答：

- 它是否真的采购过我们的目标产品；
- 搜索词是否发生误匹配；
- 邮箱或电话是否真实可用；
- 数据来自哪里，能否追溯；
- 每获得一家可联系公司花了多少钱。

本项目把这些问题放进同一条可审计的数据链路：

```mermaid
flowchart LR
    A[目标产品与严格匹配规则] --> B[海关公司搜索 API]
    B --> C[data/raw 原始付费响应]
    C --> D[本地严格产品过滤]
    D --> E[按 company_id 全局去重]
    E --> F[公司联系方式 API]
    F --> G[邮箱和电话状态验证]
    G --> H[官网与相关人物补救]
    H --> I[company-master.json]
    I --> J[唯一 Excel 客户总表]
    J --> K[后续合规邮件触达]
```

其中 `K` 是下一阶段，当前仓库没有发送邮件。

## 3. 已实现的功能

### 3.1 采购商搜索

- 使用 `POST /agent/customs/company/list` 搜索采购商。
- 支持单词、精确短语、同义词和翻页游标。
- 按贸易次数排序，优先处理更有采购证据的公司。
- 全球检索，欧洲和北美优先。

### 3.2 严格产品匹配

API 返回后还要在本地二次过滤，而不是直接入表。例如：

- `masking film` 不能误收录 `thick film` 电子元件；
- `PVC 护角条` 必须同时满足 PVC、墙角和护条语义；
- `塑料桶` 必须同时满足塑料与 bucket/pail；
- `油漆刷` 必须是 paint/painting 与 brush 构成同一产品表达，不能因为一段描述里分别出现油漆和牙刷就入选。

严格规则和回归测试位于 `scripts/expand_to_target.py` 与 `tests/test_incremental_expansion.py`。

### 3.3 联系方式获取与验证

- 使用 `POST /agent/customs/company/contact/batch` 批量获取公司邮箱、电话、网站和社交媒体。
- 每批最多 20 个公司 ID。
- 邮箱只有状态 `1` 才算有效；`0` 是未能检测，`2` 是无效，`3` 是不确定。
- 电话只有记录为 `状态1` 才算有效，并保留号码类型与 WhatsApp 状态。
- 当公司 API 不足时，可以使用官网、政府文件、权威目录和相关人物联系方式补救，但必须记录来源。

### 3.4 公司级去重和业务指标

- Excel 一行只代表一个 `company_id`。
- 同一家公司可以命中多个产品品类，但不会因此产生多行。
- 业务 KPI 以“公司是否可联系”为单位，不以邮箱或电话号码数量为单位。
- 原始邮箱条数、去重邮箱数和公司—邮箱关联数只用于技术审计。

### 3.5 成本和安全控制

- 所有付费脚本默认 dry-run。
- 必须显式添加 `--execute` 才会请求付费 API。
- 当前累计费用上限为 ¥500；已使用 ¥387.10，剩余 ¥112.90。
- 请求前按保守价格预检，请求后按响应中的 `fee.apiCost` 记录实际费用。
- API Key 只从 `UPKUAJING_API_KEY` 环境变量读取，不进入 Git。

### 3.6 可追溯 Excel 交付物

唯一总表为 `deliverables/客户表（7.17）.xlsx`，包含三个工作表：

| 工作表 | 用途 | 演示时重点 |
|---|---|---|
| 客户总表 | 270 家公司的业务主数据 | 筛选产品、欧美优先级、邮箱状态；查看公司、联系方式和贸易证据 |
| 成本与流程 | 公司级 KPI、费用构成和流程规则 | 展示获客成本、覆盖结构和审计口径 |
| 字段说明 | 每个字段的人话解释 | 回答“这一列是什么意思、来自哪里” |

## 5. 现场应该怎么讲

### 5.1 30 秒版本

> 我们不是从网上随机抓邮箱，而是先从海关贸易数据中找真正采购过目标产品的公司，再用严格规则排除错品，之后才购买和验证联系方式。现在已经沉淀了 270 家一行一公司的客户主表，所有公司至少有一个有效邮箱或电话；全链路保留原始证据、验证状态和成本记录，可以复查，也能继续扩展新品类。

### 5.2 三分钟展示顺序

1. 打开 GitHub 首页，说明仓库公开、代码和数据版本可追溯。
2. 打开 Excel 的“客户总表”，先展示 270 家总数和八个品类。
3. 筛选“产品品类（中文）= 油漆刷”，展示严格命中的公司和报关品名证据。
4. 筛选“市场优先级 = 高-欧美”，说明欧美优先但全球补足。
5. 展示“邮箱可用状态”“电话验证详情”，说明只有状态 `1` 才计入 KPI。
6. 打开“成本与流程”，展示 ¥1.43/家和累计 ¥387.10。
7. 打开“字段说明”，证明每列都有来源和解释。
8. 回到仓库，展示 `data/raw/`、`company-master.json`、测试和 dry-run 设计。

### 5.3 现场不要做的事情

- 不要现场运行带 `--execute` 的命令；这会产生真实费用。
- 不要声称系统已经自动发送邮件；当前没有发送模块。
- 不要声称 270 家都有有效邮箱；正确说法是 270 家都有邮箱或电话，205 家有有效邮箱。
- 不要把公司、邮箱、电话三个维度的数量混为一谈。
- 不要为了演示直接运行全部历史处理脚本；这可能降级主数据。

## 6. 正确表述与错误表述

| 推荐说法 | 不准确说法 | 原因 |
|---|---|---|
| 基于真实贸易记录筛选采购商 | 从网上批量爬邮箱 | 数据来源和价值完全不同 |
| 270 家公司都有至少一种有效联系方式 | 270 家都有有效邮箱 | 实际有效邮箱公司为 205 家 |
| 邮箱状态 1 才计入有效邮箱 | 表里有邮箱就能发 | 状态 0/2/3 均不能计入有效指标 |
| 每家有效联系方式公司成本 ¥1.43 | 三项获客成本加起来 | 三项只是同一费用的不同分母 |
| 当前完成批量邮件前置数据工程 | 已经可以自动群发并处理退订 | 发送闭环尚未实现 |
| 一家公司可关联多个品类但只占一行 | 各品类数量相加等于公司总数 | 品类会重叠 |

## 7. 新同事如何在本地开始

### 7.1 克隆和安装

```bash
git clone https://github.com/Stephen-creater/upkuajing-tape-masking-film-leads.git
cd upkuajing-tape-masking-film-leads

python3 -m venv .venv
source .venv/bin/activate
pip install -e .
```

项目的 Python 核心只使用标准库，无需数据库或 Web 服务。它是一个 Git 管理的数据流水线和 CLI，不是需要部署到服务器才能访问的网站。

### 7.2 先做无费用验证

```bash
PYTHONPATH=src python3 -m unittest discover -s tests -v
python3 scripts/expand_to_target.py status
upkuajing-leads
```

预期结果：

- 测试全部通过；
- 状态显示主表 270 家、本轮油漆刷 20/20、累计费用 ¥387.10；
- `upkuajing-leads` 只输出费用预估并提示 dry-run，不会请求 API。

### 7.3 API Key

需要执行付费任务时，由项目负责人通过 Git 之外的安全方式提供 Key：

```bash
export UPKUAJING_API_KEY='由负责人单独提供'
```

不要把 Key 写入代码、README、提交记录、截图或聊天转发文件。

### 7.4 Excel 重建环境

```bash
node scripts/build_workbook.mjs
```

生成器依赖 `@oai/artifact-tool`。当前项目是在提供该依赖的 Codex 工作区中生成 Excel；普通 Node 环境不一定自带此包。若接手同事只需要展示，直接下载仓库中已生成的 Excel 即可。若要维护生成器，应先确认有可用的 Codex 工作区依赖，不能假设普通 `npm install` 后就能重建。

## 8. 新增一个产品品类的标准流程

以下流程涉及真实费用，必须先获得产品定义、数量目标和预算授权。

1. 明确中文品类、英文搜索词、必须同时出现的语义和排除项。
2. 在 `scripts/expand_to_target.py` 增加产品配置、目标数量和严格匹配规则。
3. 在 `tests/test_incremental_expansion.py` 添加正例和反例。
4. 先运行全部测试，再运行不带 `--execute` 的预演。
5. 每次只搜索一页，检查严格命中率和公司重复情况。
6. API 原始响应先写入 `data/raw/`，再更新候选池。
7. 候选量略高于目标后，按欧美优先顺序分批购买联系方式。
8. 每批后检查有效公司数；达到目标立即停止，不购买剩余候选。
9. 新公司经营范围转换为中文，API 原文仍保留。
10. 更新 `company-master.json`、成本指标、README 和交接文档。
11. 重建唯一 Excel，检查公式错误、中文字段和渲染预览。
12. 运行测试，按小步 `commit + push`，最后确认本地 `HEAD` 等于 `origin/main`。

当前油漆刷任务已经达到 20/20，候选池中仍有 1 家未购买候选。不要继续执行 `contacts --execute`，否则会产生费用但不会增加目标数量。

## 10. 数据字段怎么理解

最重要的字段包括：

- `公司ID`：跨境魔方公司唯一 ID，也是去重主键；
- `产品品类（中文）`：公司明确命中的目标品类，可多选；
- `市场优先级`：欧美优先或全球常规；
- `首选邮箱`、`邮箱可用状态`：业务联系入口及验证结论；
- `首选电话`、`电话验证详情`：电话号码、有效状态、类型和 WhatsApp 状态；
- `报关品名/命中证据（原文）`：证明为什么把公司列入该品类；
- `匹配贸易次数`、`贸易总次数`、`最近贸易日期`：采购相关性和活跃度；
- `经营范围（中文）`：便于业务人员理解，原文保存在 JSON 和 raw 证据；
- `搜索原始来源`、`API 联系方式来源`、`官网核验来源`：追溯入口；
- `跟进状态`、`负责人`：销售团队后续人工维护字段。

完整解释在 Excel 的“字段说明”工作表。

## 11. 质量和安全原则

### 11.1 数据质量

- 绝不编造缺失联系方式；
- 不把不同公司 ID 静默合并；
- 不把状态 0 或 3 的邮箱重复盲测；
- 不因产品词在不同上下文偶然共现就判定命中；
- 公司名、联系方式、网址、地址和贸易描述保留原文；
- 面向业务的国家、品类、搜索词和经营范围使用中文。

### 11.2 费用安全

- 所有付费操作默认 dry-run；
- 新任务必须有明确预算；
- 先搜索、再本地过滤，最后才购买少量联系方式；
- 每达到目标立即停止；
- 以 API 响应中的真实费用为准，不只相信价目表。

### 11.3 Git 规则

- 每个可验证小步都要提交和推送；
- 原始证据不覆盖；
- 不创建 `final-v2`、`new` 等平行 Excel；
- 结束时检查 `git status` 为空，且本地 `HEAD` 等于 `origin/main`。

## 12. 当前边界和已知限制

1. **没有正式邮件发送模块。** 尚未实现模板、发送队列、发送账号、退信、回复、退订和抑制名单。
2. **不是完整的 raw-to-master 一键重建。** 部分官网和人物补救含人工审查结果，不能把所有历史脚本直接串起来覆盖主表。
3. **邮箱状态 1 不是送达保证。** 正式发送仍要通过小批量投递验证硬退信和软退信。
4. **API 价格可能变化。** 文档价与实扣价曾有差异，运行前必须重新核价。
5. **Excel 生成器有工作区依赖。** 普通 Node 环境可能缺少 `@oai/artifact-tool`。
6. **公开仓库包含业务数据。** 这是当前项目的明确选择，但 API Key 永远不能进入仓库。

## 13. 下一阶段建议

如果公司要把“客户数据工程”推进到真正的批量邮件系统，建议按以下顺序建设：

1. 邮件模板和多语言变量；
2. 发送域名、邮箱账号和发件信誉配置；
3. 小批量发送队列、限速和失败重试；
4. 硬退信、软退信、回复和退订事件回收；
5. 永久抑制名单，禁止再次发送无效或退订地址；
6. 公司级发送状态回写到主表或独立数据库；
7. 按国家/地区落实 GDPR、CAN-SPAM 等合规要求；
8. 用真实送达率、回复率和商机率替代“验证状态”作为后续核心指标。

## 14. 常见问题

### 为什么不用搜索引擎直接找邮箱？

因为本项目先证明公司与产品存在真实贸易关系，再找联系方式；公司相关性比邮箱数量更重要。

### 为什么还有只有电话、没有有效邮箱的公司？

有效联系方式可以是邮箱或电话。当前有 65 家公司只有有效电话，它们仍是可联系客户，但不应进入纯邮件发送名单。

### 为什么产品品类数加起来超过 270？

同一家公司可能同时采购塑料桶和油漆刷等多个品类，但在主表仍只占一行。

### 为什么不继续验证状态 0 或 3？

重复请求可能再次付费但没有新增信息。更有效的方式是查新官网证据、找相关人物，或在正式发送阶段用真实退信闭环验证。

### 现在能直接开始群发吗？

不能直接大规模群发。应先选择有状态 1 邮箱的公司，准备合规模板、退订机制和抑制名单，再小批量发送。

### 接手后第一天应该做什么？

1. 阅读本文件和 README；
2. 下载并浏览 Excel 的三个工作表；
3. 克隆仓库并运行测试、状态和 dry-run；
